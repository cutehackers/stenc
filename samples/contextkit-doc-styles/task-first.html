Provider Batch Contract

This document explains how one provider request can carry multiple independent parse jobs while the parser still owns message semantics, schema validation, and review decisions.

Superpowers spec header

Goal
Define the provider batch behavior without creating a second parser contract.
Architecture
One generated page renders structured JSON sections in a fixed order.
Tech Stack
ContextKit JSON, dependency-light renderer, repository validation scripts.

Reading path

Start with the summary, confirm the scope, then read the contract rules. This page favors explanation over density.

The reader is learning the system for the first time or needs a reliable explanation before touching code. The page should feel like a service manual: direct, linear, and hard to misread.

Scope

In

  • Batch request envelope and item correlation.
  • Provider failure recording.
  • Human review handoff fields.

Out

  • Provider pricing decisions.
  • Prompt tuning experiments.
  • UI styling outside generated docs.

Architecture flow

  1. 01Read source JSON metadata and links.
  2. 02Render human and agent summaries first.
  3. 03Render scope, contracts, surfaces, validation, and handoff.

Contract rules

  1. Every request item keeps a stable correlation key.
  2. Provider failures are recorded without discarding valid siblings.
  3. Human review sees the exact source message and parse result.
  4. Generated operation rows never hide review-only parser output.
