ContextKit Authoring Contract

ContextKit documents are readable web pages backed by one structured JSON source. The page makes human summary, source of truth, file surfaces, validation, and agent instructions equally visible while keeping the Superpowers section order intact.

Status: Draft Owner: ContextKit Updated: 2026-05-19

Superpowers template header

For agentic workers

Use this document as the source of truth before implementation. Follow tasks in order and prove each change with validation.

Goal
Define a fixed ContextKit document style based on Superpowers specs and plans.
Architecture
Render one structured JSON document into stable sections: summary, goal, architecture, scope, tasks, surfaces, validation, handoff.
Tech Stack
ContextKit JSON, generated static HTML, plain CSS, Node validation scripts.

Human and agent summaries

Human summary

Use this page to decide whether a document is safe to read, review, and execute.

Agent summary

Preserve the JSON source contract and update templates, examples, validator, and renderer together.

Scope

In

  • Superpowers-style document header.
  • Stable task, validation, and handoff sections.
  • API-style reference components only where useful.

Out

  • Markdown or MDX as source documents.
  • Per-document visual components.
  • Renderer behavior that depends on a docs framework.

Implementation tasks

task 1

Align the JSON contract

Keep required fields identical across templates, references, examples, and validator.

  • Update templates.
  • Update examples.
  • Run validation.
task 2

Render fixed sections

Render sections in the same order on desktop and mobile.

  • Render header and scope.
  • Render tasks and validation.
  • Verify responsive order.

API-inspired component

Document reference

schema v1
GET /context-kit/{collection}/{slug}

This does not turn ContextKit into an API reference. It borrows the useful API-doc pattern of putting operation, inputs, outputs, and errors in stable positions.

Parameters

NameTypeRequiredDescription
collectionstringYesDocument family such as specs or plans.
slugstringYesStable document route identifier.
sourcepathYesJSON file that owns the contract.

Example output

{
  "status": "valid",
  "route": "/specs/example-runtime/",
  "sourceOfTruth": ["content/specs/example-runtime.json"]
}
Permission local repository access
Error shape field, reason, fix

Decision record

accepted

Use a fixed renderer

Authors provide structured JSON. The generated page decides the visual structure so every document is predictable.

constraint

Keep install light

The renderer must not depend on a docs framework, MDX, frontmatter, or per-document visual components.
