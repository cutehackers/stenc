# Rail Artifact Identity Contract

<Aside type="danger" title="Canonical Contract">
This example shows how a Rail contract can be represented with ContextKit.
</Aside>

## Human Summary

Rail work is identified by artifact handles, not request files or inferred task
names. Users should be able to resume or inspect work through durable artifacts.

## Agent Summary

When changing artifact identity, update runtime contracts, tests, and any skill
surface that describes start, resume, retry, status, or result behavior.

## Source Of Truth

The canonical source of truth remains the Rail product spec. This example is a
ContextKit style sample, not active Rail product authority.

## Problem

Artifact identity rules are easy to blur when plans mention request files,
target-local state, or generated output. That can make agents infer the wrong
resume or projection authority.

## Scope

### In

- Artifact handle identity
- Resume and retry semantics
- Result projection authority

### Out

- Downstream application behavior
- Provider-specific event formats

## Non-Goals

- This example does not replace Rail's canonical docs.
- This example does not define new runtime behavior.

## Vocabulary

| Term | Meaning |
| --- | --- |
| Artifact handle | Durable identity returned by Rail for task or goal work |
| Projection | Status or result read from persisted artifacts |

## Contract

The system must preserve these rules:

- Fresh work creates a fresh artifact handle.
- Existing work resumes only from a validated handle.
- Result projection reads artifacts only.

## Interfaces

### Public Surface

```python
rail.start_task(draft)
rail.supervise(handle)
rail.result(handle)
```

### Internal Surface

- Artifact store
- Supervisor
- Evaluator gate
