# ContextKit Base Template

<Aside type="note" title="Template Basis">
Start here for structure. Then use the matching spec, plan, decision, or agent-context template for the final document type.
</Aside>

## Human Summary

Write the document so a human can scan the status, scope, risks, and validation
without reading every paragraph.

## Agent Summary

Write the document so an AI coding agent can identify the source of truth,
owned surfaces, execution order, validation commands, and stop conditions
without guessing.

## Source Of Truth

Name the canonical spec, plan, decision, code module, product surface, external
document, or artifact that owns the behavior.

## Goal

State what this document is meant to settle, guide, or preserve in one or two
sentences.

## Architecture

Describe the shape of the work or contract. Include the primary components,
boundaries, data flow, or decision path only when they matter to future agents.

## Scope

### In

- Name the exact behavior, files, APIs, workflows, or docs covered by this page.

### Out

- Name adjacent work that this page does not authorize.

## Non-Goals

- List tempting but intentionally excluded changes.

## Evidence

List the real inputs behind the document: code paths, artifacts, user decisions,
runtime output, review notes, or external references.

## Agent Instructions

<Steps>
1. Read the source of truth before editing related implementation or docs.
2. Preserve required frontmatter and headings.
3. Keep behavior, scope, evidence, and validation visible in Markdown.
4. Stop and surface conflicts instead of reconciling contradictory sources silently.
</Steps>

## Review Checklist

- A completed document has no unresolved `TBD` outside `Open Questions`.
- The source of truth and owned surfaces are explicit.
- Validation commands are copy-pasteable.
- Starlight components improve scanning only; they do not hide the contract.
