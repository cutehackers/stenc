Rail Artifact Identity Contract

Example single-document Stenc spec for Rail artifact identity and projection authority.

Status: draft Owner: rail-runtime Updated: 2026-05-19

Human Summary

Rail work is identified by artifact handles, not request files or inferred task names. Users should be able to resume or inspect work through durable artifacts.

Agent Summary

When changing artifact identity, update runtime contracts, tests, and any skill surface that describes start, resume, retry, status, or result behavior.

Goal

Show how one durable runtime contract can name identity rules, owned surfaces, validation, and agent instructions in one JSON document.

Architecture

Rail separates request drafting, artifact persistence, supervision, and result projection. Artifact identity belongs to the persisted artifact layer, while supervision and result projection consume validated handles.

  • Draft request
  • Create artifact handle
  • Supervise handle
  • Project result from artifacts

In

Out

  • Provider-specific event formats

Problem

Artifact identity rules are easy to blur when plans mention request files, target-local state, or generated output. That can make agents infer the wrong resume or projection authority.

Requirements

REQ-1

Durable artifact identity

Rail work must be identified by a persisted artifact handle rather than by request file paths or inferred task names.

Acceptance Criteria

  • Fresh work creates a fresh artifact handle.
  • Resume and retry flows require a validated existing handle.

Approaches

Artifact-handle contract

Tradeoffs

  • Requires explicit handle validation
  • Prevents agents from guessing resume authority from nearby files

Recommendation

Keep artifact handles as the only resume and result-projection authority.
